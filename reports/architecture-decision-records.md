# ORVION Architecture Decision Records (ADR)

Version: 0.1
Status: Living document (not a Change Request; preserves the reasoning behind major architectural decisions)
Convention: append-only. Each ADR is numbered and dated. A superseded ADR is marked Superseded and points to its replacement; it is never deleted (history is the record). New ADRs are added when a genuinely architectural decision is made — routine implementation choices stay in their Change Request.

---

## ADR-0001 — Platform: PostgreSQL on Supabase
- Date: 2026-07 · Status: Accepted · Source: PROJECT_CONTEXT.md §5
- Decision: ORVION's database is PostgreSQL, operated through Supabase (Auth, Data API, Storage, local CLI stack).
- Why: managed Postgres with built-in auth/RLS/storage matches a SaaS CRM's needs without building that infrastructure; local CLI gives reproducible migrations.
- Consequences: migrations authored for the Supabase CLI; auth integrates via the `auth` schema; RLS is the tenant-isolation mechanism.

## ADR-0002 — UUID primary keys via `gen_random_uuid()`
- Date: 2026-07 · Status: Accepted · Source: 30 Primary Key Standard; SPEC-022
- Decision: every main table uses `id uuid primary key default gen_random_uuid()`; `pgcrypto` is enabled first.
- Why: globally unique, non-guessable, generatable client- or server-side; avoids sequence contention in a multi-tenant/distributed future.
- Consequences: keys are immutable (supports `on update no action`); reference tables may instead use a stable natural key (see ADR-0010).

## ADR-0003 — Shared-schema multi-tenancy via `tenant_id` + RLS
- Date: 2026-07 · Status: Accepted · Source: 30 Tenant Scope + RLS Standards
- Decision: one schema; every tenant-owned table carries `tenant_id uuid not null references tenants(id)`; isolation enforced by RLS (Tenant → Branch → Department).
- Why: shared-schema scales to many tenants with low operational overhead; RLS enforces isolation in the database, not just the app.
- Consequences: RLS policies are required on every tenant table (planned migration 19); `tenants` is the isolation root.

## ADR-0004 — `users` links to `auth.users` via a separate `auth_user_id`
- Date: 2026-07 · Status: Accepted · Source: 30 Identity Key Standard; 31 §13 item 3; SPEC-009
- Decision: `users` keeps its own `id`; a separate `auth_user_id uuid references auth.users(id)` (nullable, **unique per tenant** — `unique (tenant_id, auth_user_id)`) links the membership to Supabase Auth. `users.id` is NOT set equal to `auth.users.id`. `auth_user_id` is null while invited-but-not-activated, and set on activation. See ADR-0011 for the membership model.
- Why: keeps `users.id` stable and provider-independent (every other table FKs to it); allows a business user to exist before its auth row; leaves room for a second identity provider without a breaking PK migration.
- Consequences: RLS resolves `auth.uid()` → business user through `auth_user_id`; authorization stays in ORVION RBAC, not JWT claims.

## ADR-0005 — Catalog-based lookups over native enums
- Date: 2026-07 · Status: Accepted · Source: 25_catalog_registry.md; 30 Catalog/Status Standards
- Decision: dropdown/lookup values live in `catalog_types` / `catalog_values` (composite `(catalog_type_code, code)`), not PostgreSQL `enum` types. Stable public datasets use dedicated reference tables instead (ADR-0010).
- Why: enums are hard to alter and cannot hold tenant-specific values; catalog tables support system + tenant-extendable values and are queryable.
- Consequences: `catalog_values` needs the composite UNIQUE; codes are seeded (migration 18) and governed by 25.

## ADR-0006 — Status/type codes are plain text; DB enforcement optional per-column
- Date: 2026-07 · Status: Accepted · Source: 30 Status Standard (resolved); SPEC-030
- Decision: status/type code columns are plain `text`, validated by the seeded catalog + application/state-machine logic; a foreign key is not required. Hard DB enforcement (validation trigger, or constant type column + composite FK) is optional per column.
- Why: a single code column cannot reference `catalog_values`' composite key, and these codes are written by application code / state machines, not free-typed by users. Code stability (never renamed/deleted) keeps stored values valid for audit/history.
- Consequences: replaces the earlier un-implementable "composite FK is the canonical strategy" wording; each status column maps to one family registered in 25.

## ADR-0007 — Referential actions default to `on delete restrict on update no action`
- Date: 2026-07 · Status: Accepted · Source: 30 Referential Action Standard; SPEC-027
- Decision: every FK defaults to `on delete restrict on update no action`; `cascade`/`set null` are opt-in per FK with justification.
- Why: matches archive-not-delete philosophy (parents are archived, never physically deleted); UUID/natural keys are immutable so cascade-on-update is dead machinery.
- Consequences: deleting a referenced parent is blocked by default (proven in SPEC-029).

## ADR-0008 — `updated_at` maintained by a database trigger (`moddatetime`)
- Date: 2026-07 · Status: Accepted · Source: 30 Timestamp Standard; SPEC-027/028
- Decision: every table with `updated_at` has a `before update` trigger; `moddatetime` is the recommended mechanism (hand-written `plpgsql` allowed).
- Why: guarantees `updated_at` advances for every writer including direct SQL and the Supabase `service_role`, which bypass the application.
- Consequences: the trigger mechanism is enabled before the first trigger; each table adds its trigger in its own migration.

## ADR-0009 — Git workflow: direct-to-`main`, publish on Complete
- Date: 2026-07 · Status: Accepted · Source: Governance decision; SPEC-026
- Decision: Change Requests execute as linear commits on the current branch (direct-to-`main` in practice); branch/PR topology is intentionally undefined. Completing a CR pushes to the configured upstream.
- Why: "Git is the execution history"; the CR lifecycle already provides review gates; a linear history mirrors the state machine. Publishing on Complete removes a separately-remembered manual push.
- Consequences: no branch/PR governance is defined; a failed push does not invalidate a local Complete.

## ADR-0010 — Reference data uses stable natural keys
- Date: 2026-07 · Status: Accepted · Source: 25 (Reference Data); 31 §2a; SPEC-025
- Decision: stable public reference datasets (e.g., `currencies`) are dedicated tables keyed by their natural code (`currencies.code`), not surrogate UUIDs, and are FK targets for `*_code` columns.
- Why: the natural code is the stable, externally meaningful identifier every referencing column already uses.
- Consequences: a deliberate, documented deviation from ADR-0002 for reference tables; the reference-data layer (countries/languages/nationalities) is due before migrations 8–10.

## ADR-0011 — `users` is a tenant membership; `auth.users` is the human
- Date: 2026-07 · Status: Accepted · Source: SPEC-033; Migration 5 identity review
- Decision: a `users` row is a human's **membership/employment in one tenant**, not a global human identity. The human identity is `auth.users` (one login per person). `auth_user_id` is **unique per tenant** (`unique (tenant_id, auth_user_id)`), so one human may hold at most one membership per tenant and may hold memberships in several tenants — each a separate `users` row referencing the shared `auth.users`.
- Why: multi-tenant SaaS must support one person across several organizations (employees changing agencies, consultants/contractors, franchises, parent/multi-entity operators). Fixing this on an empty table costs one constraint; retrofitting it after ~70 tables reference `users` and RLS is built is a foundational migration. The MVP single-membership experience is behaviourally identical.
- Consequences: (1) authentication-layer facts (credentials, trusted devices, MFA, global suspension) belong to `auth.users`; business-layer facts (roles, activity, per-company status, audit) belong to the membership. (2) RLS resolves `auth.uid()` **plus an active-tenant context** to the membership (migration 19) — degrades to the single membership when a human has one. (3) Disabling a user (`is_active=false`) is company-scoped; global ban is an `auth.users` action. (4) `trusted_devices`/`otp_challenges`/`totp_enrollments` are authentication-layer and should attach to the human identity, not the per-tenant membership — a migration-16 concern (Future Backlog). Supersedes the global-uniqueness portion of ADR-0004.

## ADR-0012 — Authentication & Identity Principles are canonical; auth-support tables re-home to the human identity
- Date: 2026-07 · Status: Accepted · Source: Owner directive; SPEC-046; builds on ADR-0011
- Decision: the authentication/identity/security philosophy is consolidated into one canonical document, `34_authentication_and_identity_principles.md`, whose twelve principles govern every future auth-related decision ("which principle owns this responsibility?" precedes schema). Its direct schema consequence: `trusted_devices`, `otp_challenges`, `totp_enrollments` key to `auth_user_id` referencing `auth.users(id)` (Human Identity), carry no `tenant_id` and no membership `user_id`, and cascade on human-identity deletion. `31` §9 and `33` migration 16 are amended accordingly.
- Why: the philosophy was previously distributed across ADR-0011, `20_authentication_security_model.md`, and backlog notes, forcing the "where does this table belong?" question to be re-litigated per table. A principles document lets the schema fall out as a consequence. Re-homing follows Principles 1/6/7: device trust, OTP, and MFA prove *who the human is* and are established before tenant selection, so a person must not re-trust devices or re-enrol MFA once per company.
- Consequences: (1) auth-support RLS (migration 19) is row-ownership by `auth.uid()`, with no tenant scoping — simpler than tenant-scoped policies. (2) the *requirement* to present a factor stays role-driven through RBAC, decoupled from the artifact's storage. (3) migration 16 depends only on the Supabase-provided `auth.users`, not on `users`. Closes the migration-16 Future-Backlog item.

## ADR-0013 — Tenant Isolation & Data Access Principles are canonical (consolidate RLS decisions)
- Date: 2026-07 · Status: Accepted · Source: Owner directive; SPEC-050; consolidates ADR-0011, `30` Tenant Scope + Identity Key Standards, `31 §13` item 3
- Decision: the tenant-isolation/RLS philosophy is consolidated into one canonical document, `35_tenant_isolation_and_data_access_principles.md`, governing Migration 19. It restates the settled decisions (database-enforced default-deny isolation by `tenant_id`; `SECURITY DEFINER` RBAC resolution, never JWT claims; MVP degrades to the single membership) and adds the derivable engineering rule that **every table policy references one resolution primitive** so the resolution mechanism evolves in one place, plus: helper functions are `SECURITY DEFINER` in a non-API schema; `catalog_values` is readable when `tenant_id is null` OR = resolved tenant; platform/support access is the `service_role` (RLS-bypassing) backend, not per-table policy; audit logs are append-only (insert + tenant-scoped select, no update/delete).
- Why: the RLS decisions were scattered (ADR-0011, two `30` Standards, `31 §13`), forcing "where does this rule live?" per table — the same situation auth had before Migration 16. A principles doc lets the policies fall out as a consequence and prevents ~70 policies from hardcoding resolution.
- Consequences: (1) subscription-state gating is kept **distinct** from tenant isolation (authority = `subscription_status`, not `tenants.status`; enforced at service layer for MVP or a later separate predicate). (2) the deferred F2 FKs (`catalog_values.tenant_id`/`created_by`) should land before/with Migration 19. (3) multi-membership active-tenant plumbing, when it ships, touches only the resolution function. No genuinely unresolved question blocks Migration 19.

## ADR-0014 — Backend architecture is Supabase-native-first (no separate backend service)
- Date: 2026-07 · Status: Accepted · Source: Owner decision; backend architecture evaluation
- Decision: ORVION's backend is **Supabase-native-first**. PostgREST + RLS + PostgreSQL functions (RPC) are the API and business-logic backbone; **Edge Functions** (Deno/TS) + `pg_cron`/`pg_net` + n8n handle out-of-database compute (external API calls, webhooks, schedulers, AI calls); the web app is a server-rendered frontend (`@supabase/ssr` + `supabase-js`) talking directly to Supabase; future mobile and public/external API consumers use the same Supabase surface. **No standalone application server** is introduced; a dedicated service (or API gateway) is added later only when a concrete capability provably outgrows this — an explicit, documented trigger, not a pre-emptive layer.
- Why: authorization already lives in the database (RLS + `SECURITY DEFINER` resolution; ADR-0011/0013, Migration 19), so a separate backend would have to re-implement and could contradict that single source of truth. Every client type (web, mobile, external API — the schema already seeds `ACCESS_API_READ_ONLY`/`ACCESS_API_FULL`) inherits consistent enforcement through the same surface. Lowest ops/cost for a small owner-driven SaaS; the integration roadmap (Phase 10) is already Edge-Function-shaped; confirms the existing `@supabase/ssr` signal. Rejected alternatives: a dedicated backend service (duplicates DB-resident authz, adds ops/cost, chokepoint for all clients — kept only as a future targeted addition); pure PostgREST with no Edge Functions (can't do out-of-DB integration work); a frontend-framework BFF as the core (works for the web client only, not mobile/API); a full custom stack (discards Supabase Auth + the RLS investment).
- Consequences: minimal ops and low fixed cost; one authz model across all clients; near-free new client types; integrations land as isolated Edge Functions; the portable asset (Postgres schema + RLS) is preserved. Watch-items with documented future triggers: (a) a dedicated **API gateway** when public APIs / partner integrations / third-party developers arrive; (b) an **AI orchestration/gateway** layer only when evidence shows real value; (c) ongoing clarity of background-processing responsibilities across Postgres / Edge Functions / `pg_cron` / n8n / any future workflow engine to avoid overlap. Heavy/complex logic goes into TypeScript Edge Functions (testable) rather than growing large RPCs. Governs phases 3-10.

## ADR-0015 — Permission matrix realizes onto the binary `role_permissions` table (current operational model)
- Date: 2026-07 · Status: Accepted · Source: Owner decision (Option A); permission-model realization
- Decision: ORVION realizes the permission matrix (`28_permissions_matrix.md`) onto the **binary `role_permissions(role_id, permission_id)`** table exactly as defined in frozen schema 31 — **no scope/condition/plan columns are added**. The matrix's other three dimensions are enforced at point-of-use: **scope** (tenant/branch/department/assigned/platform) structurally by RLS + RPC logic that already knows the caller's tenant/branch/department/assignment; **conditional** grants (Optional / Assigned-only / Limited) inside the guarding RPC; **plan-gating** via the subscription-state / `feature_entitlements` path. Seeding rule: `role_permissions` holds only the **strict "Yes"** cells of the matrix; Optional/Assigned-only/Limited/scoped-conditional/No cells receive **no row yet** and are added when the RPC/enforcement that narrows them is built (earn-it).
- Why: `role_permissions` is a frozen-schema table and is strictly binary; scope is *already* enforced by RLS (ADR-0013), so duplicating it as grant columns would create a second, divergent source of truth. Encoding "Optional/Limited" as columns now bakes an unreviewed interpretation before any consumer exercises those semantics. Keeps the schema as simple as possible and lets implementation earn additional structure. Rejected alternatives: enriching `role_permissions` with `scope_code`/`condition_code` (+ plan gate) — alters a frozen table and pre-encodes future assumptions; deferring `role_permissions` entirely — the owner chose to make the binary model the operative one now, not to defer it.
- Consequences: role→permission grants are a clean binary set; permission checks (`has_permission()`, when built) read this table and combine it with RLS scope + RPC conditional logic + subscription gating. **This is the current operational model, not a permanent limitation.** Escalation trigger: if a future real capability demonstrates that scope-aware / conditional / plan-aware permissions can no longer be expressed cleanly through RLS + RPCs + business logic, raise it as a **new architectural decision with evidence** rather than extending the schema preemptively; likewise, present a cleaner/more-standard industry approach with evidence and await owner approval before changing direction. Seeded by SPEC-057.

## ADR-0016 — Tenant provisioning is platform-mediated (current operational model)
- Date: 2026-07 · Status: Accepted · Source: Owner decision (Option A); tenant-provisioning model
- Decision: New tenants and their first owner user are **provisioned by the platform**, not via self-serve signup. `app.provision_tenant(...)` is a `SECURITY DEFINER` RPC callable **only by `service_role`** (platform owner / backend); it creates the tenant, the owner `users` membership (linked to a Supabase `auth.users` id), and an owner `user_role_assignments` row (`scope_type='tenant'`), returning both ids. The tenant's initial `status` is a parameter defaulting to `'trial'`. Provisioning deliberately does **not** create a `subscriptions` row or seed `subscription_plans`/`feature_entitlements` — the subscription/billing lifecycle is a separate, later slice.
- Why: matches ORVION's current business model — bank-transfer billing with platform-owner subscription approval/renewal (`09_saas_plans_and_access.md`), not online self-serve payments. Keeping tenant creation a controlled `service_role` operation avoids introducing a self-serve billing/security surface the model doesn't yet support, and resolves the bootstrap gap (RLS requires a `users` membership that could not otherwise be created). Rejected alternatives: self-serve signup (introduces a public provisioning + billing path the bank-transfer model doesn't support, widens the security surface); seeding tenants only by migration (no app-driven onboarding path).
- Consequences: the first owner exists and can authenticate; subsequent user/branch/department/role management RPCs build on this owner. `app.has_permission(permission_key)` (the binary authorization primitive per ADR-0015, resolving the caller within `app.current_tenant_id()` through active `user_role_assignments` → `role_permissions`) is introduced alongside provisioning and verified end-to-end against the provisioned owner. **Current operational model, not a permanent limitation.** Escalation trigger: if the business model evolves (self-service onboarding, online payments/Stripe, automated subscriptions, public sign-up), present a new architectural decision with evidence before changing the provisioning model. Deferred (recorded, not half-built): the "Role assigned" security_event mandated by `28`'s Event Requirements, wired when the eventing capability is built. Implemented by SPEC-058.

## ADR-0017 — Authentication artifacts belong to Supabase Auth; ORVION owns authentication policy (Supabase-native authentication)
- Date: 2026-07 · Status: Accepted · Source: Owner decision (Option A); authentication-hardening architecture
- Context / evolution: ORVION originally anticipated **owning** the authentication artifacts — the migration-29 tables (`totp_enrollments`, `otp_challenges`, `trusted_devices`) and `34_authentication_and_identity_principles.md` (Principle 7) were designed with ORVION storing MFA/OTP secrets and enrolment. After **ADR-0014** established the Supabase-native architecture, responsibility for authentication **artifacts** (passwords, MFA/TOTP secrets and enrolment, OTP challenges, sessions, identity assurance level) **intentionally moved to Supabase Auth (GoTrue)**, while ORVION **retained ownership of authentication policy, authorization, business security rules, and role-based enforcement**. This is an architectural evolution, not an obsolescence: the earlier schema recorded a genuine intent that the backend decision then re-homed.
- Decision — two principles: (1) **Authentication artifacts belong to the Identity Provider (Supabase Auth).** The TOTP factor + secret live in `auth.mfa_factors`; enrolment, challenge, verification, and email-OTP are performed by Supabase Auth, which stamps the session assurance level (`aal1` = single-factor, `aal2` = MFA-verified) into the JWT. ORVION does not store or verify MFA/OTP secrets (no rolling-our-own MFA). (2) **Authentication policy and business authorization belong to ORVION.** ORVION owns *which* roles require MFA (the high-risk roles `owner`/`ceo`/`finance_manager`/`system_administrator` per `28`) and enforces the **requirement** — exactly Principle 7's "the artifact is global to the human; the policy that triggers it is scoped to the membership." Enforcement primitives: `app.requires_mfa()` (does the caller's active membership hold a high-risk role) and `app.mfa_satisfied()` (true unless required and the session is not `aal2`, read from the JWT `aal` claim); sensitive RPCs assert it. `trusted_devices` remains an ORVION-owned concept (Supabase has no first-class trusted-device primitive); device-trust records key to `auth.users` per Principles 1/6.
- Consequences: single source of truth for the MFA secret (Supabase Auth); no re-implemented crypto; canon `28`'s "high-risk roles require TOTP" is enforced via the `aal` claim. The migration-29 `totp_enrollments`/`otp_challenges` tables are **not used to store secrets** — under the Supabase-native model that responsibility is Supabase Auth's; the frozen tables remain in place (schema frozen at 31) and may be repurposed as thin policy/projection stores or retired in a future documented change, not now. Principle 7's schema consequence in `34` is reinterpreted accordingly (the "Human Identity" that owns the secret is Supabase Auth). The high-risk role set is transcribed from `28`; making it data-driven (a `roles` column) is a deferred refinement (schema is frozen). Implemented by SPEC-062.

## ADR-0018 — Background processing: business logic in RPCs, scheduling by the fittest scheduler, external I/O in Edge/n8n (scheduler-agnostic)
- Date: 2026-07 · Status: Accepted · Source: Owner decision; first scheduled workload (lead SLA). Resolves ADR-0014's deferred "background-processing responsibilities" trigger.
- Decision (a **scheduler-agnostic** principle, deliberately separating architecture from implementation): (1) **Business logic** lives in **PostgreSQL RPCs** — idempotent, self-contained, and directly testable with controlled inputs. (2) **Scheduling** is performed by **the most appropriate scheduler for the current maturity level**; *today that is `pg_cron`* (in-DB, Supabase-native, minimal ops). pg_cron is the current *implementation*, not the *principle* — if a better scheduler later fits, the architecture is unchanged because the logic is not coupled to the scheduler. (3) **External I/O** (email/WhatsApp delivery, webhooks, third-party calls) lives in **Edge Functions and/or n8n**; in-system effects (a `notifications` row, an `events` row) are in-DB and need no external scheduler.
- Why: the detection/action logic of scheduled work (e.g. "which assigned leads are overdue → warn/reassign/notify") is pure data work that belongs next to the data, stays transactional, and is testable without waiting on wall-clock time (parameterise the thresholds). Keeping the scheduler as a thin, replaceable trigger avoids coupling and premature infrastructure. Rejected alternatives: a standalone scheduler/worker service (unearned ops/cost); embedding schedule logic in an Edge Function as the primary home (splits logic from data, harder to test, needed only for external I/O); making pg_cron itself the architectural principle (couples architecture to one tool).
- Consequences: the first scheduled workload — lead SLA (SPEC-067) — is `app.process_lead_sla()` (a `service_role`/`SECURITY DEFINER` RPC doing detection → `lead_sla_warning` + in-system notification at the warn threshold → reassignment + `lead_reassigned` at the reassign threshold), scheduled every minute by pg_cron; the RPC is tested deterministically by backdating assignment timestamps, independent of the schedule. External notification *delivery* (email/WhatsApp) stays deferred to an Edge Function/n8n until that capability is earned. **Escalation trigger — re-evaluate the scheduling layer only when a workload genuinely needs**: distributed queues, retry orchestration, cross-service workflows, high-volume asynchronous processing, or orchestration beyond straightforward scheduled DB jobs. Until then, keep it as simple as possible (RPC + pg_cron). Establishes the pattern for all future scheduled work (subscription/document/passport expiry, SLA timers). Implemented by SPEC-067.

## ADR-0019 — Customer-merge re-points all references via dynamic FK discovery (participation-by-default)
- Date: 2026-07 · Status: Accepted · Source: Owner decision (dynamic FK-driven re-pointing over a static allowlist); customer-identity merge design.
- Decision: `app.merge_customer_identity(source, target, reason)` **discovers every table/column that references `customers(id)` from the PostgreSQL system catalogs (`pg_constraint`) at run time** and re-points each from the source customer to the target, rather than maintaining any manual list of participating tables. Discovery is **inclusion-by-default with a small, explicit, documented exclusion list** — currently only `customer_identity_merges` (the audit record of merges themselves, which legitimately references both source and target). The merge writes a `customer_identity_merges` row (source, target, `merged_by`, reason, timestamp), emits the mandated `customer_merged` event (`28` — customer merge is sensitive and must create an event), and archives the source customer (soft, never physical delete). It is `SECURITY DEFINER` (so re-pointing is complete regardless of the caller's RLS row visibility) with `app.authorize('MERGE_CUSTOMER_IDENTITY')` + MFA enforced at entry and explicit same-tenant verification of both customers; re-pointing keys on the globally-unique customer `id`, so it never crosses tenant boundaries.
- **Architectural rule for future contributors:** *Every new table that references `customers` automatically participates in customer-merge operations unless it is explicitly excluded for a documented reason (added to the merge's exclusion list with a comment).* This makes merge-completeness a property of the platform architecture, not of developer memory — a new referrer cannot silently split a customer's history.
- Why: `customers(id)` is already referenced by ~14 tables, most in future phases (bookings, invoices, payments, passengers, quotations, complaints, service_requests, conversations, refunds). A static allowlist would become a latent correctness bug the moment any future table adds `customer_id` and is forgotten — producing orphaned/split customer history that is hard to detect and hard to reverse. Catalog-driven discovery stays correct across schema evolution with zero maintenance. Rejected alternatives: static inclusion list (maintenance hazard, silent incompleteness); `SECURITY INVOKER` under RLS (a future restrictive policy could hide rows from the caller and silently skip re-pointing them); physical delete of the source (destroys business history, violates the never-delete principle).
- Consequences: merges are complete-by-construction and fully auditable (merge row + immutable `customer_merged` event give traceability without needing undo). Escalation trigger: if a future referrer must **not** be blindly re-pointed (e.g. it needs value-level conflict resolution, deduplication, or a merge-time transformation rather than a straight FK swap), add it to the documented exclusion list and handle it explicitly in the merge — do not weaken the default. Implemented by SPEC-070.

## ADR-0020 — Finance-approval gate: execution-approval slice now; capability-driven booking permissions recorded, minted per-consumer; negative-balance risk flag deferred to Finance Core
- Date: 2026-07 · Status: Accepted · Source: Owner decision after the Phase-5 Finance Gate Readiness Report (`reports/phase-05-finance-gate-readiness.md`). First Phase-5 ADR.
- Decision: the Finance Approval Gate (`13`, `26` Finance Approval State Machine) is built as an **execution-approval** control on the generic booking item, keyed to the item's finance fields (`finance_approval_required`, `finance_approval_status_code`, `cost_locked_at`) and the existing `approval_requests(approval_type_code='finance_execution_approval')` — **never to a service type**, so Finance stays service-agnostic. It is delivered in slices: (1) `app.request_finance_approval` opens a `pending` request and marks the item; (2) `app.review_finance_approval` (`APPROVE_FINANCE`) approves/rejects/records the decision and, on approve, locks cost; (3) the already-marked `confirmed → in_progress` edge in `app.advance_booking_item` gains a **precondition** (an approved finance approval must exist when the item requires it) — an additive change, no rework. The Finance Approval State Machine is honoured exactly: `pending → approved|rejected|cancelled`, `rejected → pending` (resubmit); events are the seeded `finance_approval_requested|approved|rejected|cancelled|resubmitted` (`27`).
- **Permission model (capability-driven, per `booking-transition-authority-model`):** booking authority is **capability-driven** (Submit / Approve / Issue / Cancel / Refund / Reissue), not a single generic booking-status permission — this ADR **records that model as the decided direction**. Applying **Earn-It / "seed only strict-Yes cells" (ADR-0015)**, permissions are **minted by the CR that first consumes them**, not up front. Consequently the execution-approval slice mints **zero** new permissions: `request` reuses `CREATE_BOOKING_ITEM` (the booking operator withdraws/raises approval on an item they already manage), `review` uses the existing `APPROVE_FINANCE`, and gating the execution edge stays under `UPDATE_BOOKING_ITEM_STATUS` (the gate adds a precondition, not an authority). The finance-gated **booking-level** transitions (`pending_approval → confirmed` [Approve], issuance [Issue], void/refund/reissue [Cancel/Refund/Reissue]) and their specific permissions land with those transitions in later CRs.
- **Explicitly deferred — negative-balance risk flag:** `13`/`28` require that issuing before full collection records a risk flag capturing *customer balance*. There is no `customer_balances` table (balance is derived from `invoices`/`payments`/`refunds`) and no balance primitive yet. This depends on a **Finance Core** `app.customer_balance(customer_id[, booking_id])` and is deferred until Finance Core provides it; at that point the risk flag is emitted as a first-class **event** (no table), per the event philosophy. `ALLOW_ISSUE_WITH_NEGATIVE_BALANCE` already exists and is reserved for the issuance slice, not this one.
- Why: the execution-approval control is fully buildable against the current schema and is the safe, additive core of the gate; splitting it out ships value without waiting on Finance Core. Recording the capability model but minting per-consumer avoids seeding unused permission rows (a latent drift/audit hazard) while still fixing the direction so future transitions don't re-litigate it. Keying the gate to the generic booking item preserves the service-agnostic Booking Core (`booking-orchestration-boundary`). Rejected alternatives: a single generic `UPDATE_BOOKING_STATUS` permission (loses capability granularity, `28` has no such perm); minting the whole Submit/Issue/Cancel/Refund/Reissue set now (seeds permissions with no consumer — violates Earn-It and ADR-0015); building the negative-balance risk flag now against a non-existent balance primitive (would fabricate a balance concept in the wrong layer).
- Consequences: Booking Core takes only additive changes. Design-Challenge findings folded in: `request` blocks a duplicate `pending` request and requests on a terminal/archived item or booking; `finance_approval_status_code` is written only with values from the `approval_status_code` catalog (`pending/approved/rejected/cancelled`) for consistency (`§3D` of the readiness report). Escalation triggers: (a) mint `REQUEST_FINANCE_APPROVAL` only if operational experience shows request authority must diverge from `CREATE_BOOKING_ITEM`; (b) build the negative-balance risk flag when `app.customer_balance()` exists; (c) the quotation↔lead↔booking gap (`§3B`) remains a separate recorded capability, not part of this gate. Implemented by SPEC-081 (`request_finance_approval`) and the following finance-gate CRs.

## ADR-0021 — Customer balance is a derived, per-currency, read-only primitive
- Date: 2026-07 · Status: Accepted · Source: Phase-6 Finance Core first capability; forward-referenced by ADR-0020 (negative-balance risk flag dependency).
- Decision: `app.customer_balance(p_customer_id uuid, p_booking_id uuid default null)` is the single authoritative definition of a customer's outstanding balance. It is **derived** (never stored) from live records: per currency, `outstanding = invoiced − paid + refunded`, where invoiced = live customer invoices (`status_code in issued/partially_paid/paid/overdue`; not voided, not archived), paid = `customer_payment` payments, refunded = **completed** `customer_refund` refunds. Positive = customer owes the company; negative = credit/overpayment. Results are grouped **by currency** and never collapsed across currencies (multi-currency is real; collapsing requires exchange rates). Optional `p_booking_id` narrows to one booking. Read-only (no writes, no events), `stable`, `security invoker`, tenant-scoped via `app.current_tenant_id()`; RLS on `invoices`/`payments`/`refunds` is the only access gate — no `app.authorize()`, following the read-RPC precedent (`app.lead_booking_readiness`). Future consumers (outstanding-balance reporting; the ADR-0020 negative-balance risk flag) MUST call this function, never re-derive the rule.
- Why: derived-not-stored eliminates drift between a cached balance and the finance records (matches the Phase-5 readiness framing). Per-currency is the only correct aggregation without an exchange-rate policy, which is a separate future concern. Counting only completed refunds and live (non-draft/non-void) invoices reflects the actual receivable/cash position. No finance-read permission was minted: the permission model is action-oriented and read RPCs rely on RLS (Earn-It — do not invent a `VIEW_FINANCE` permission without a consumer needing finer control). Rejected alternatives: a stored/materialized balance (drift hazard; needs invalidation on every finance write); a single cross-currency scalar (silently wrong under multi-currency); gating with a new permission (unearned permission-model expansion; RLS already scopes the underlying rows).
- Consequences: unblocks the deferred negative-balance risk flag (ADR-0020) and outstanding-balance reporting (Phase 9). Finer-grained finance-read authorization, if ever needed, is an RLS-policy decision on the finance tables, not a change to this function. Cross-currency reporting in a single presentation currency is a separate capability layered on `exchange_rates`, leaving this primitive unchanged. Implemented by SPEC-089.

## ADR-0022 — Phase 9 read-model & reporting architecture (security_invoker views + RLS-bearing aggregate tables; NO API-exposed materialized views)
- Date: 2026-07-17 · Status: Accepted (technical architecture; one isolated owner business-policy decision — presentation currency — is deferred, see the Owner-decision bullet). Source: Phase 9 (Reports & Dashboards / RC-4) architecture program. Learn-Before-Designing evidence: Supabase docs + PostgreSQL RLS current practice (2026), cited below.
- Context: Phase 9 delivers operational visibility (lead performance, sales activity, booking pipeline, outstanding balances, profit by booking item, subscription state). All source data is built (Phases 2–7) and the derived read primitives already exist (`app.customer_balance`, `app.supplier_balance`, `app.booking_item_profit`, `app.lead_booking_readiness`). ORVION is multi-tenant with RLS on all 71 tables; tenant is resolved via `app.current_tenant_id()` (JWT-derived, not a session GUC — pooling-safe).

- Decision (the read-model ecosystem):
  1. Dedicated `reporting` schema holds every read-model object. Keeps reporting isolated from the transactional `public`/`app` surface (One Authority; clean grants; expose/lock as a unit).
  2. Two-tier materialization, chosen per query cost:
     - Tier A — live `security_invoker` views (Postgres 15+, `with (security_invoker = true)`) over transactional tables + the existing read primitives, for cheap/low-volume reports. The view runs with the caller's permissions, so base-table RLS is automatically enforced — no separate policy needed (and none is possible) on the view. This is the default; prefer it.
     - Tier B — tenant-scoped aggregate tables (ordinary tables carrying `tenant_id` + their own RLS policies) for expensive rollups (e.g., daily pipeline/finance aggregates). Refreshed out-of-band. The aggregate table is the cache.
  3. NO materialized views on any API-reachable path. Proven constraint: materialized views do not support RLS (Supabase pg-meta classifies `relkind = 'm'` as "RLS not supported"; PostgreSQL confirms matviews carry no row-security), so an exposed matview would leak across tenants. Aggregate tables (which do bear RLS) replace them. (A matview is permissible only if never API-exposed and queried solely by a trusted server-side path — not used here.)
  4. Refresh = `SECURITY DEFINER` RPC invoked by `pg_cron` (scheduler-agnostic per ADR-0018 — logic in the RPC, scheduling in the fittest scheduler). Each aggregate has an idempotent `reporting.refresh_<name>(p_tenant uuid default null)` that recomputes and upserts tenant-tagged rows (all tenants, or one). `SECURITY DEFINER` is required to read across tenants during refresh; it writes only tenant-tagged rows and never returns data to a caller.
  5. External BI (Superset/Metabase, future) connects through a dedicated non-`BYPASSRLS` role so RLS still filters every query; BI is a consumer of the same secured surface, never a privileged bypass.

- The 13 program dimensions (as designed):
  - Reporting strategy: view-first (Tier A); promote to Tier B only when measured cost warrants (Earn-It — do not pre-aggregate cheap reports).
  - Dashboard layer: the six outputs are `reporting` views/aggregates exposed via PostgREST (auto-REST over the `reporting` schema) or read RPCs; the same objects feed external BI. No bespoke API service (ADR-0014 Supabase-native).
  - Materialization strategy: Tier A live; Tier B pre-aggregated tables; matviews excluded from exposed paths (RLS gap).
  - Refresh strategy: `pg_cron` → `SECURITY DEFINER` refresh RPC per aggregate; cadence per report (default nightly for daily rollups; tighter only where earned). Incremental where a natural watermark exists (`created_at`/`metric_date`), full-recompute otherwise.
  - Security model: Tier A inherits base RLS via `security_invoker`; Tier B tables carry `tenant_id` + RLS (`tenant_id = app.current_tenant_id()`); the refresh RPC is the only `SECURITY DEFINER` element and returns nothing. RLS is the gate — no new `VIEW_*` permission is minted unless a consumer needs finer control (Earn-It, per ADR-0021).
  - Tenant isolation: every read-model object is tenant-scoped and RLS-enforced; the matview exclusion removes the one bypass vector; external BI role is non-`BYPASSRLS`.
  - Aggregation strategy: expensive group-bys (pipeline by stage, finance by currency/period, profit rollups) precomputed into Tier B; cheap current-state reads stay Tier A over the primitives.
  - Caching strategy: the Tier B aggregate tables ARE the cache — no separate cache system (Earn-It; a second cache is drift + ops cost). PostgREST/edge response caching may be layered later on measured need.
  - Performance strategy: index the RLS predicate (`tenant_id`) and each report's group-by/filter columns on aggregate tables; Tier A relies on existing base-table indexes (tenant_id indexes landed, SPEC-114). Refresh RPCs `analyze` touched aggregates.
  - Observability: a `reporting.refresh_runs` log table (tenant nullable, `aggregate_name`, `started_at`, `finished_at`, `rows_written`, `status`, `error`) records every refresh; failures also emit an `events`/`security_events` entry.
  - Monitoring: `cron.job_run_details` + `reporting.refresh_runs` give job health; a staleness check (max `finished_at` per aggregate vs expected cadence) is the alert signal.
  - Operational maintenance: refresh RPCs are idempotent (safe re-run); aggregate tables get `analyze` post-refresh and standard autovacuum; cadence lives in one config table (`reporting.refresh_schedule`), not scattered cron literals.

- Owner business-policy decision (ISOLATED — does not block the rest): presentation currency. Reports summing money across currencies need a policy: (A) per-currency (default — matches every built primitive, which is already per-currency per ADR-0021; zero new dependencies; the strongest technical default) or (B) single presentation currency (requires an FX conversion capability: a `reporting.convert_amount(amount, from_ccy, to_ccy, at)` `stable` function selecting the latest `exchange_rates.rate` with `effective_at <= at` per tenant, plus a rate-source/rounding policy). This is an owner decision because it changes what the business sees, not how the system works. Design proceeds on (A) per-currency; option (B) is additive and non-breaking — it layers `convert_amount` over `exchange_rates` (already present: tenant-scoped, temporal, `numeric(18,8)`) without touching Tier A/B structure. Nothing else in Phase 9 waits on this.

- Rejected alternatives: API-exposed materialized views (RLS bypass → cross-tenant leak — disqualifying); a separate caching service (Redis/etc. — unearned ops cost + drift vs the DB, Earn-It); storing balances/profit (drift hazard — ADR-0021 settled derived-not-stored); a bespoke reporting API service (violates ADR-0014 Supabase-native); minting `VIEW_REPORTS` permissions up front (unearned — RLS already scopes rows).
- Consequences: Phase 9 implements as — create `reporting` schema → Tier A `security_invoker` views over the four primitives + transactional tables for the six outputs → add Tier B aggregate tables + refresh RPCs + `pg_cron` + `refresh_runs`/`refresh_schedule` only where a report's cost earns it. FX (option B) is a clean later slice if the owner selects single-currency. Phase 8 later consumes this same verified-outcome surface (offline-conversion value), satisfying the Revenue-Intelligence posture (PROJECT_CONTEXT §11) — built once, consumed by humans and ad platforms alike. Supersedes nothing; extends the read-primitive line (ADR-0021).
- Learn-Before-Designing sources: Supabase docs — Row Level Security (`security_invoker` views), Tables/View security; Supabase pg-meta RLS-support classifier (matviews & foreign tables unsupported). PostgreSQL 18 §5.9 Row Security Policies. Current multi-tenant RLS practice (2026): index RLS predicates; reset tenant context per pooled request (N/A here — JWT-derived, not GUC).

## ADR-0023 — Offline-conversion delivery: Google Data Manager API + Enhanced Conversions for Leads, orchestrated by n8n over a database outbox
- Date: 2026-07-17 · Status: Accepted (owner-ratified — compliance posture + transport both approved). Source: `reports/history/google-offline-conversion-transport-decision-2026-07-17.md` (full option analysis; Learn-Before-Designing 2026 research). Supersedes nothing; consumes ADR-0022 (read-model as the verified-outcome surface) and ADR-0018 (logic in RPCs, orchestration external).
- Owner decisions ratified: (1) ORVION sends verified conversion outcomes to Google — hashed (SHA-256, normalized) email/phone + GCLID/GBRAID/WBRAID + consent signals, delivering only when consent = granted. This closes the founding feedback loop (Google Ads → GTM → GA4 → CRM → verified offline conversion → Google). (2) **n8n is the orchestration layer** — confirmed as ORVION's primary automation engine for Google, Meta, WhatsApp Cloud API, email, and future integrations.
- Decision (architecture):
  1. **Transport:** Google **Data Manager API** with **Enhanced Conversions for Leads**. The legacy Ads-API `UploadClickConversions` path is unavailable to ORVION (blocked for new integrations 2026-06-15).
  2. **Division of responsibility (n8n-first):** ORVION owns *truth and state*; n8n owns *transport and orchestration*. All conversion logic — outcome mapping, consent gating, delivery state, retry bookkeeping — lives in ORVION RPCs. n8n is a **stateless caller**: it claims work, calls Google, reports the result. If n8n dies mid-run, no state is lost — unacknowledged claims become claimable again.
  3. **Database outbox pattern:** `offline_conversions` + `offline_conversion_deliveries` form the outbox. ORVION exposes an n8n-facing RPC pair: `app.claim_conversion_deliveries(p_batch int)` (atomically claims pending, consent-granted conversions; creates delivery rows `pending`, returns the delivery payload incl. identifiers to hash) and `app.record_conversion_delivery_result(p_delivery_id, p_status, p_response, p_error)` (records `sent`/`failed`; failed rows become re-claimable per the retry policy — statuses are the seeded `pending/sent/failed/retried`). Granted to the integration role only, never to `authenticated`.
  4. **Consent gate in ORVION, not n8n:** a conversion is claimable only when its attribution click carries `consent_ad_user_data = 'granted'` (SPEC-119 columns). The compliance rule is enforced in the database, so no orchestration bug can ship an unconsented row.
  5. **Hashing at the edge:** raw email/phone leave the database only to n8n over the private connection; n8n normalizes + SHA-256-hashes before Google. Google never receives raw PII.
  6. **Channel-generic boundary:** the claim/ack outbox is not Google-specific — `offline_conversion_deliveries.platform_code` (seeded catalog) keys the destination, so Meta CAPI (Phase 10) reuses the identical mechanism with a different n8n workflow. One delivery model, many platforms.
- Why: n8n-first was re-evaluated against the existing design (owner directive 2026-07-17): the transport-agnostic adapter boundary survives unchanged — only the *caller* changes from an Edge Function to n8n workflows. The outbox + claim/ack pattern is the standard reliable-delivery design for external orchestrators (at-least-once with idempotent acks); it makes n8n swappable (any orchestrator — or a future Edge Function — can drive the same RPC pair), preserving ORVION-owns-the-truth (PROJECT_CONTEXT §11). Rejected: queue infrastructure (pg_mq/external broker — unearned; the deliveries table IS the queue at ORVION's volume, escalation trigger per ADR-0018 stands); Edge-Function-internal delivery logic (would bury state where the confirmed orchestrator cannot see it); per-platform delivery tables (duplicate structure; platform_code already discriminates).
- Consequences: Phase 8 implements — conversion mapping (verified outcome → `offline_conversions` row; value = revenue, never profit — margins are not leaked to ad platforms; industry-standard) → the claim/ack RPC pair + integration role → the n8n workflow (Data Manager API + OAuth `datamanager` scope + Consent Mode). GTM/GA4 client-side tagging coexists via Google's unified enhanced-conversions setting. The Integration Catalog (future-backlog, Adopt-Later) seeds with this integration when Phase 8 lands.
